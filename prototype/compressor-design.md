# Designing a compressor

> A compressor is not one algorithm. Building one means choosing a detector, a topology, a
> knee, a place to smooth, ballistics, lookahead, and makeup — and each choice changes the
> sound. This page maps the decision space and holds a full implementation that exposes
> every choice.

*Chapter 5 — the design space behind [Compression](compression.md), and the implementation
that plays it.*

## The decision map

Designing a compressor is a series of choices. The map below starts from no design; picking
one option at each decision traces a path, and every path ends at a complete compressor.
The toggles overlay five open-source implementations — each is a different path through the
same decisions, and no two take the same route. Context on the five projects is in the
[References](references.md) appendix.

Two facts the map understates:

- Not every path composes. A feedback topology cannot look ahead: the output it would need
  to inspect does not exist yet. The implementation below rejects the combination.
- The map's resolution is itself a design choice. The options listed are the coarsest set
  that still separates all five reference implementations; a coarser map would make
  distinct designs look identical.

## The implementation

`code/compressor.py` is a configurable compressor whose keyword arguments are the rows of
the map. One engine plays every valid path:

```python
compress(x, sr, detector="rms", topology="feedforward", knee="soft_quad",
         smoothing="after", ballistics="fixed", lookahead="true", makeup="manual")
```

The five reference implementations are included as `PRESETS`, so
`compress(x, sr, **PRESETS["audacity"])` runs their path. The module docstring records
where the code narrows the map and why. Program-dependent ballistics is described but not
implemented.

The engine is exercised by `code/test_compressor.py` (`python3 -m unittest discover -s
code`): unity below threshold, exact ratio arithmetic, ceiling guarantees with and without
lookahead, and a check that all five presets produce different output on the same material.
The time-domain figures in this chapter are generated by running this engine
(`code/make_figures.py`).
